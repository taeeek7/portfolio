# n8n 워크플로우 자동화

> 코드 배포 없이 운영 요청을 처리하는 로우코드 워크플로우 체계
> 프로젝트: `integrations/n8n`

---

## 프로젝트 개요

외부 트리거(Webhook·구글시트)를 받아 **어드민 API를 호출하고 결과를 Slack으로 알리는** 운영 자동화 워크플로우 모음입니다.

파트너사 교체에 따른 대량 일감 재세팅, 시트 기반 티켓 일괄 생성, 개발 환경 세팅 등
**"코드로 만들 만큼은 아니지만 손으로 하기엔 너무 반복적인"** 작업을 로우코드로 처리합니다.

| 항목 | 내용 |
|------|------|
| **워크플로우** | 10+ (live/test 이중화) |
| **런타임** | n8n Cloud |
| **연동** | 어드민 API, Google Sheets, Slack, DataTable |

---

## 1. 문제 정의

**문제 1 — 운영 요청마다 코드를 배포할 수는 없다**

- "이 5개 지점 파트너사가 바뀌었으니 일감을 다시 세팅해달라" 같은 요청은 **일회성이지만 규모가 큼**
- 매번 스크립트를 짜서 배포하기엔 리드타임이 길고, 손으로 하기엔 실수 위험

**문제 2 — 자동화가 늘수록 운영 환경에서 검증 없이 돌게 된다**

- 운영 데이터를 건드리는 자동화를 테스트 없이 돌리면 **되돌리기 어려운 사고**로 이어짐

**문제 3 — n8n의 워크플로우가 UI에만 존재하면 이력이 남지 않는다**

- 누가 언제 무엇을 바꿨는지 추적 불가, 리뷰 불가

---

## 2. 문제 해결

### 2-1. ★ 핵심 원칙 — live/test 이중화

**모든 워크플로우를 live/test 두 벌로 유지**하는 것을 원칙으로 세웠습니다.

```
workflows/
├── live-apply-settings.json        ┐
├── live-clear-settings.json        │ 운영 (indicator.{domain})
├── live-create-ticket-by-gsheet.json ┘
├── test-apply-settings.json        ┐
├── test-clear-settings.json        │ 테스트 (indicator.test.{domain})
├── test-create-ticket-by-gsheet.json │
├── test-dedup-cleanup.json         │
├── test-message-routing.json       │
└── test-current-settings-dump.json ┘
```

**환경별로만 달라지는 항목을 명시적으로 정의**해, 이것 외에는 로직이 동일함을 보장했습니다.

| 동기화 제외 항목 | 이유 |
|------------------|------|
| API 도메인 | live / test 엔드포인트 분리 |
| 인증 credential | 환경별 권한 격리 |
| Slack Webhook URL | 알림 채널 분리 |
| DataTable 환경 필터 | live는 `env: "live"` 조건 부가 |

> 로직을 수정하면 **양쪽에 모두 반영**하되, 차이는 위 4가지에 한정합니다.
> 운영 데이터를 건드리는 자동화이므로, test에서 먼저 검증한 뒤 live에 적용하는 흐름을 강제한 것입니다.

### 2-2. 워크플로우를 Git으로 관리

n8n UI에만 존재하면 이력이 남지 않으므로, **JSON을 export해 Git으로 버전 관리**하고
CLI 스크립트로 배포합니다.

```bash
# workflows.conf: 워크플로우명 → n8n 인스턴스 ID 매핑
scripts/deploy.sh <command> [args]
```

- `.env`에서 인증 정보를 로드하고 `workflows.conf`로 ID를 매핑해, **JSON 파일이 곧 배포 단위**가 됨
- 변경 이력·리뷰가 Git에 남음

### 2-3. n8n 특유의 함정을 문서로 고정

n8n JSON은 직접 편집할 때 깨지기 쉬운 지점이 많아, **주의사항을 CLAUDE.md에 명문화**했습니다.

| 함정 | 규칙 |
|------|------|
| `node id` | 워크플로우 내 고유 UUID, n8n이 내부 참조 → **임의 변경 금지** |
| `connections` | 노드 **이름**으로 연결됨 → 이름 변경 시 `connections` 블록과 Code 노드의 `$('노드이름')` 참조도 함께 수정 |
| `typeVersion` | 인스턴스 호환성 문제 → 임의로 올리지 않음 |
| `splitInBatches` | output 0 = 루프 완료, output 1 = 반복 실행 (직관과 반대) |
| `If` 노드 | output 0 = true, output 1 = false |
| `credential ID` | export된 인스턴스 기준 → 다른 인스턴스 import 시 UI에서 재선택 필요 |

> 이 함정들은 **한 번 겪고 나서야 아는 것들**이라, 문서로 남겨 반복 학습 비용을 없앴습니다.
> AI 코딩 도구에도 이 규칙을 학습시켜 워크플로우 수정 시 자동 준수하게 했습니다.

## 3. 아키텍처

```
┌──────────────────┐   ┌──────────────────┐
│  Webhook 트리거   │   │  Google Sheets   │
│  (외부 시스템)    │   │  (운영자 입력)    │
└────────┬─────────┘   └────────┬─────────┘
         │                      │
         └──────────┬───────────┘
                    ▼
        ┌───────────────────────────┐
        │      n8n Cloud            │
        │                           │
        │  ┌─────────────────────┐  │
        │  │ Loop (splitInBatches)│  │
        │  │   └ If 분기          │  │
        │  │   └ Code 노드 변환   │  │
        │  │   └ DataTable 조회   │  │
        │  └──────────┬──────────┘  │
        └─────────────┼─────────────┘
                      ▼
        ┌───────────────────────────┐
        │  어드민 API               │
        │  (일감 생성·설정 적용·초기화)│
        └─────────────┬─────────────┘
                      ▼
        ┌───────────────────────────┐
        │  Slack 결과 알림           │
        │  (환경별 채널)             │
        └───────────────────────────┘
```

### 주요 워크플로우

| 워크플로우 | 용도 |
|------------|------|
| `create-ticket-by-gsheet` | 구글시트 입력 → 티켓 일괄 생성 |
| `apply-settings` / `clear-settings` | 공간·키퍼그룹 설정 일괄 적용 / 초기화 |
| `current-settings-dump` | 현재 설정 상태 덤프 (변경 전 백업) |
| `dedup-cleanup` | 중복 데이터 정리 |
| `message-routing` | 메시지 조건부 라우팅 |

### 실제 활용 사례

- **파트너사 교체 5개 지점 일감 재세팅** — 설정 덤프 → 초기화 → 재적용 순으로 워크플로우 조합
- **개발 환경 공간·키퍼그룹 자동 세팅** — 테스트 환경 구성 반복 작업 제거

## 4. 기술 스택

| 영역 | 기술 |
|------|------|
| **워크플로우 엔진** | n8n Cloud |
| **주요 노드** | Webhook, HTTP Request, Code(JS), If, splitInBatches, Google Sheets, Slack, DataTable |
| **배포** | zsh 스크립트 (`deploy.sh`) + n8n API |
| **버전 관리** | Git (워크플로우 JSON export) |
| **연동 대상** | 어드민 API, Google Sheets API, Slack Webhook |

---

## 5. 성과 / 효과

### 리드타임

- 일회성 대량 운영 요청을 **코드 배포 없이 처리** → 요청부터 처리까지의 리드타임 단축
- Airflow DAG로 만들 만큼 정기적이지 않은 작업을 **로우코드로 흡수**

### 안전성

- live/test 이중화로 **운영 데이터 변경 전 검증 강제**
- `current-settings-dump`로 변경 전 상태를 남겨 **롤백 가능성 확보**
- 환경별 credential·알림 채널 분리로 **오적용 방지**

### 유지보수

- 워크플로우 JSON을 Git으로 관리해 **변경 이력·리뷰 가능**
- n8n 함정을 문서로 고정해 **반복 학습 비용 제거**, AI 도구에도 동일 규칙 적용

---

## 6. 회고

**잘한 판단**

**live/test 이중화를 원칙으로 못박은 것**. 로우코드 도구는 UI에서 바로 실행할 수 있어 편한 만큼,
검증 없이 운영 데이터를 건드리기 쉽습니다. "같은 로직 두 벌 유지"는 관리 비용이 들지만,
운영 사고 한 번의 비용에 비하면 훨씬 쌉니다.

**남은 과제**

- 같은 로직을 두 벌 유지하는 구조라 **동기화 누락 위험**이 상존 (자동 diff 검증 미구현)
- 워크플로우 수가 늘면 n8n UI에서의 탐색성이 떨어짐
