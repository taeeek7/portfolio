# AI 에이전트 생태계 - 사내 반복업무 위임 플랫폼

> 역할이 분리된 서브에이전트 16종 + 도메인 스킬 20여 종 + 자체 MCP 서버
> 프로젝트: `.claude/{agents,skills,mcp-servers}`

---

## 프로젝트 개요

기획·개발 과정의 반복업무(배포 공지 작성, 정책 반영 분석, 번역 키 생성, 데이터 분석, 티켓 관리 등)를
**AI 에이전트에 위임 가능한 형태로 구조화**한 사내 도구 플랫폼입니다.

단순히 AI를 "쓰는" 것이 아니라, **역할·권한·책임이 분리된 에이전트 체계를 설계하고 운영 회고를 통해 개선**해 왔습니다.
설계 결정과 폐기 사유를 결정 로그로 남겨, 무엇을 왜 그렇게 했는지 추적 가능하게 관리합니다.

| 항목 | 내용 |
|------|------|
| **서브에이전트** | 16종 (secretaries 8 / code-advisors 4 / product-advisors 4) |
| **도메인 스킬** | 21종 (사내 워크플로우 자동화) |
| **자체 MCP 서버** | mail-mcp, reminders-mcp + [Keeper Agent](../11-keeper-agent/) |

## 1. 문제 정의

**문제 1 — AI에게 맡길 때마다 컨텍스트를 다시 설명해야 한다**

- "우리 서버 레포는 Spring Boot 멀티모듈이고, 정책은 여기 있고, 커밋 컨벤션은 이렇고…"를 매번 반복
- 결과적으로 AI 활용이 **일회성 질의**에 머물고 축적되지 않음

**문제 2 — AI가 의도치 않은 쓰기 작업을 한다**

- 코드 이해를 물었는데 파일을 고치거나, 슬랙 메시지를 확인만 하려는데 전송하는 위험
- **읽기와 쓰기의 권한 경계가 없으면** 안심하고 위임할 수 없음

**문제 3 — 컨텍스트 오염**

- 무거운 원문(긴 Slack 스레드, 대용량 HTML, gh CLI 출력)을 메인 대화에 그대로 가져오면 정작 중요한 맥락이 밀려남

## 2. 문제 해결

### 2-1. ★ 핵심 설계 — 역할에 따른 3계층 분리

에이전트를 **무엇을 할 수 있는가**로 나누지 않고, **무엇을 해서는 안 되는가**로 나눴습니다.

```
┌──────────────────────────────────────────────────────────────┐
│  secretaries/ — 외부 도구 executor                            │
│  읽기 자유 / 쓰기는 dry-run 미리보기 후 사용자 승인            │
│  jira · slack · github · gmail · obsidian · browser · chrome  │
├──────────────────────────────────────────────────────────────┤
│  code-advisors/ — 코드 레포 read-only 학습                    │
│  이름은 engineer지만 코드 작성·수정 안 함 (Edit/Write 미보유)  │
│  backend · web-front · app-front · data-engineer              │
├──────────────────────────────────────────────────────────────┤
│  product-advisors/ — 도메인 advisor                           │
│  정책·스키마·디자인 관점 자문                                  │
│  planner · designer · data-analyst · localizer                │
└──────────────────────────────────────────────────────────────┘
```

**공통 설계 원칙**

1. **Advisor는 read-only** — Read/Bash/Grep/Glob + 도메인 MCP만 보유, Edit/Write 없음
2. **Secretary의 쓰기는 전부 dry-run 후 승인** — 생성/수정/삭제/전송 예외 없음
3. **폴더 구조는 조직화용** — invocation 식별자는 frontmatter `name`

> `code-advisors`에 "engineer"라는 이름을 붙이면서도 **쓰기 도구를 주지 않은 것**이 의도적 설계입니다.
> 코드를 이해하는 역할과 고치는 역할을 분리해야, "물어보기만 했는데 고쳐놨다"는 사고가 구조적으로 불가능해집니다.

### 2-2. 컨텍스트 격리 — 원문은 흡수하고 digest만 반환

Secretary는 무거운 원문을 **자기 컨텍스트에서 소화**하고 메인 대화에는 요약만 돌려줍니다.

```
메인 대화 ──▶ slack-secretary ──▶ 긴 스레드 원문 200개 메시지 (격리된 컨텍스트)
                    │
                    └──▶ "핵심 3줄 + 결정사항 + 액션아이템"만 반환
```

같은 원리로 `chrome-secretary`(대용량 HTML/JSON), `github-secretary`(gh CLI 출력)를 두었습니다.

### 2-3. 도메인 스킬 — 워크플로우의 절차화

반복되는 사내 절차를 스킬로 고정해, 매번 방법을 설명하지 않아도 되게 했습니다.

| 스킬 | 자동화한 업무 |
|------|---------------|
| `release-notice` | 배포 전 KP 티켓 → 연결 PR·정책서 참조 → 사용자 공지 초안 |
| `policy-sync` | 배포 후 머지된 PR 분석 → 정책서 반영 필요 항목 식별 |
| `prototype-to-i18n-keys` | 프로토타입 코드 + 정책서 → i18n 키 생성 → 시트 업로드 |
| `figma-to-lokalise-keys` | Figma 디자인 → UI 텍스트 추출 → Lokalise 키 |
| `handover` / `resume` | 세션 인수인계 및 컨텍스트 복구 |
| `commit` / `pr` | 변경사항 분석 → 기능 단위 커밋 / PR 문서 생성 |
| `obsidian-log` / `obsidian-organize` | 지식 아카이빙 및 Inbox 분류 |

> `release-notice`(배포 전)와 `policy-sync`(배포 후)처럼 **책임 경계를 명시적으로 분리**해,
> 스킬끼리 서로의 영역을 침범하지 않도록 했습니다.

### 2-4. 자체 MCP 서버

| [`keeper-agent`](../11-keeper-agent/) | 사내 지식(정책·스키마·API·Enum·화면 IA) 조회 |

## 3. 운영 회고 기반 개선 (결정 로그)

설계 결정과 **폐기 사유**를 README에 결정 로그로 남깁니다. 실패한 시도를 지우지 않고 이유와 함께 보존합니다.

### 사례 1 — Bootstrap 자동 `git pull` 룰 폐기 (2026-05-18)

4개 advisor에 "24시간마다 레포를 자동 pull" 하는 bootstrap을 넣었다가 **제거**했습니다.

| 관찰 | 판단 |
|------|------|
| 캐시 stamp는 4개 생성됨 (스크립트는 돌았음) | 겉으로는 정상 |
| 그런데 ff 가능한 상태인데 HEAD가 안 움직임 | `pull --ff-only` 실패 추정 |
| 실패 보고 형식 준수가 4개 중 2개만 | 사용자가 실패를 감지 못함 |

> **"잘 도는 것처럼 보이지만 실제론 무력한 상태"가 가장 위험하다**고 판단해,
> 자동화를 걷어내고 수동 관리로 되돌렸습니다. 불확실한 자동화보다 명확한 수동이 낫다는 결론입니다.

### 사례 2 — capture-secretary 폐기 → browser-secretary 신설 (2026-05-25)

| 문제 | 해결 |
|------|------|
| `mcp__computer-use`가 실제 마우스/키보드를 점유 → **사용자 foreground 작업이 끊김** | Playwright **격리 브라우저 컨텍스트**로 전환 |
| 데스크톱 앱 캡쳐 수요가 사실상 없음 | 웹 전용으로 범위 축소 |
| 단순 캡쳐만 가능 | 캡쳐 + 클릭/입력/폼 + 네트워크 추출 + 동작 검증으로 확장 |

`chrome-secretary`(사용자 실브라우저, read-only, 로그인 세션 재사용)와 **분업 구조**를 명확히 했습니다.

## 5. 성과 / 효과

### 위임 가능성

- 반복업무 **16종 에이전트 + 21종 스킬**로 구조화해, 매번 컨텍스트를 재설명하지 않고 작업 위임
- 배포 공지·정책 반영·번역 키 생성·데이터 분석 등 **기획 실무 워크플로우를 절차로 고정**

### 안전성

- Advisor에서 쓰기 도구를 아예 제거해 **의도치 않은 코드 변경 구조적 차단**
- Secretary 쓰기 작업 전면 dry-run으로 **되돌릴 수 없는 외부 액션(슬랙 전송, 티켓 생성) 사고 방지**
- 격리 브라우저 전환으로 **사용자 작업 중단 문제 해소**

### 컨텍스트 효율

- 무거운 원문을 secretary 컨텍스트에서 소화하고 digest만 반환 → **메인 대화 오염 방지**
- 세션 인수인계(`handover`/`resume`)로 긴 작업의 컨텍스트 연속성 확보

## 6. 회고

**잘한 판단**

에이전트를 능력이 아니라 **권한 경계로 설계**한 것. "이 에이전트는 무엇을 할 수 있나"가 아니라
"무엇을 절대 못 하게 할 것인가"를 먼저 정하니, 안심하고 위임할 수 있는 범위가 오히려 넓어졌습니다.

그리고 **폐기한 결정을 기록으로 남긴 것**. bootstrap `git pull` 사례처럼
"돌아가는 것처럼 보였지만 실제로는 무력했던" 자동화를 발견하고 걷어낸 판단은,
같은 실수를 반복하지 않기 위한 자산이라고 봅니다.

**남은 과제**

- 개인 생산성 도구로 비칠 여지가 있음 → 팀 단위 확산·공용화는 아직 제한적
- 사용량·절감 시간 등 **정량 지표 미계측**
