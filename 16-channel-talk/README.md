# 채널톡 CS 연동 - 문의 자동 라우팅 & 상담 데이터 수집

> 고객 문의를 태그 기반으로 담당 조직에 자동 라우팅하는 서버리스 파이프라인
> 프로젝트: `integrations/channel-talk`

---

## 프로젝트 개요

고객 상담 채널(채널톡)에 들어온 문의를 **Webhook으로 수신해 문의 유형별 담당 조직을 Slack에 자동 멘션**하고,
상담 종료까지의 흐름을 **스레드로 묶어 추적**하는 서버리스 연동 시스템입니다.

더불어 누적된 상담 데이터를 Open API로 수집해, **CS 응대 매크로 설계와 문의 유형 분석**의 근거로 활용합니다.

| 항목 | 내용 |
|------|------|
| **런타임** | AWS Lambda (서버리스) |
| **라우팅 규칙** | 문의 태그 11종 → 3개 조직 매핑 |
| **상태 관리** | DynamoDB (상담 ↔ Slack 스레드 매핑) |

---

## 1. 문제 정의

**문제 1 — 문의가 담당자에게 닿기까지 사람을 거친다**

- 고객 문의가 들어오면 누군가 읽고 판단해 담당 조직에 전달해야 함
- **전달 누락·지연**이 발생하고, 특히 업무 시간 외 문의는 다음 날 아침까지 방치

**문제 2 — 문의 유형에 따라 담당 조직이 다르다**

- "앱 로그인 오류"는 기획+운영, "티켓 확인 불가"는 기획+개발, "개선 제안"은 기획
- 이 매핑을 **사람이 매번 판단**하면 오전달이 생김

**문제 3 — 상담의 시작과 끝이 흩어진다**

- 문의 접수 알림과 상담 종료 알림이 별개 메시지로 오면 **맥락 추적 불가**

---

## 2. 문제 해결

### 2-1. ★ 태그 기반 조직 라우팅

문의 태그를 **담당 조직에 명시적으로 매핑**해, 사람의 판단 없이 알림이 정확한 대상에게 가도록 했습니다.

```python
TAG_MENTION_MAP = {
    "오류신고-앱/설치·로그인오류":    [PLANNING, OPERATION],
    "오류신고-앱/티켓확인불가":       [PLANNING, DEVELOPMENT],
    "오류신고-어드민/티켓누락·중복":  [PLANNING, DEVELOPMENT],
    "오류신고-어드민/금액·정산오류":  [PLANNING, OPERATION],
    "사용법문의/앱":                  [PLANNING, OPERATION],
    "개선제안/어드민":                [PLANNING],
    ...
}
```

**설계 포인트**

| 결정 | 이유 |
|------|------|
| 태그 하나에 **복수 조직** 매핑 가능 | 오류는 대체로 기획+개발 또는 기획+운영이 함께 봐야 함 |
| 매핑에 없는 태그는 **기획팀으로 폴백** | 라우팅 실패로 문의가 **아무에게도 안 가는 상황을 원천 차단** |
| 중복 멘션 제거 | 태그 여러 개가 같은 조직을 가리켜도 한 번만 멘션 |

> 폴백을 둔 것이 핵심입니다. 새 태그가 추가됐는데 매핑을 깜빡해도, 문의는 최소한 기획팀에는 도달합니다.
> **자동화의 실패 모드가 "누락"이 아니라 "덜 정확한 전달"이 되도록** 설계했습니다.

### 2-2. 상담 생명주기를 Slack 스레드로 묶기

상담 시작(`opened`)과 종료(`close`)는 **별개의 Webhook 이벤트**로 도착합니다.
이를 하나의 맥락으로 묶기 위해 **DynamoDB에 상담↔스레드 매핑을 저장**했습니다.

```
[상담 시작]
채널톡 webhook (userChat, state=opened)
    │
    ▼
Lambda ──▶ Slack chat.postMessage (신규 메시지)
    │           └──▶ thread_ts 반환
    ▼
DynamoDB: { userChatId → thread_ts }

[상담 종료]
채널톡 webhook (message, log.action=close)
    │
    ▼
Lambda ──▶ DynamoDB 조회 (userChatId → thread_ts)
    │
    ▼
Slack chat.postMessage(thread_ts=...) ──▶ 같은 스레드에 종료 알림
```

결과적으로 **하나의 상담 = 하나의 Slack 스레드**가 되어, 접수부터 종료까지의 흐름이 한자리에 남습니다.

### 2-3. 보안 및 검증

- 채널톡 페이로드의 `token`을 **환경변수와 대조 검증**해 위조 요청 차단
- Slack 토큰·DDB 테이블명 등 민감 정보는 전부 **환경변수로 주입** (코드에 하드코딩 없음)
- 로컬 테스트 환경(`local_invoke.py` + 샘플 이벤트 JSON)을 별도로 두어 **배포 전 분기·포맷 검증**

### 2-4. 상담 데이터 수집

누적 상담 데이터를 분석하기 위해 Open API 수집 스크립트를 작성했습니다.

```
GET /open/v5/user-chats           (state별 커서 페이징) → 상담 목록
    └─▶ GET /open/v5/user-chats/{id}/messages  (커서 페이징) → 메시지 전문
            └─▶ JSONL 저장
```

- **인증 정보는 macOS Keychain에서 로드** (환경변수는 폴백) → 키가 셸 히스토리·파일에 남지 않음
- **PEP 723 인라인 의존성**으로 가상환경·requirements 파일 없이 `uv run` 단독 실행
- 채널톡 API 문서 사이트가 SPA(빈 HTML 셸 + `bundle.js`)라 크롤링이 불가능한 점을 확인하고,
  **런타임이 실제로 로드하는 스펙 JSON URL을 직접 찾아** 스펙을 확보 (`fetch_spec.py`)

## 3. 아키텍처

```
┌─────────────────────┐
│  채널톡 (Channel.io) │  고객 문의
└──────────┬──────────┘
           │ Webhook (opened / close)
           ▼
┌─────────────────────┐
│  API Gateway        │
└──────────┬──────────┘
           │
           ▼
┌──────────────────────────────────────┐
│  AWS Lambda                          │
│                                      │
│  1. token 검증                        │
│  2. 이벤트 타입 분기                   │
│     ├ userChat/opened → 신규 알림     │
│     └ message/close   → 스레드 답글   │
│  3. 태그 → 조직 멘션 매핑              │
│     (미매핑 시 기획팀 폴백)            │
└─────┬──────────────────────┬─────────┘
      │                      │
      ▼                      ▼
┌──────────────┐    ┌─────────────────┐
│  DynamoDB    │    │  Slack API      │
│  상담↔스레드  │    │ chat.postMessage│
│  매핑 저장    │    │  (+ thread_ts)  │
└──────────────┘    └─────────────────┘

── 별도 배치 ──────────────────────────
┌─────────────────────┐
│ collect_userchats.py│  Open API v5
│  (커서 페이징)       │  → JSONL
└─────────────────────┘  → 문의 유형 분석 / 응대 매크로 설계
```

## 4. 기술 스택

| 영역 | 기술 |
|------|------|
| **컴퓨팅** | AWS Lambda (Python) |
| **API 게이트웨이** | AWS API Gateway |
| **상태 저장** | AWS DynamoDB |
| **외부 API** | 채널톡 Open API v5, Slack Web API |
| **수집 스크립트** | Python + httpx, PEP 723 인라인 의존성 (`uv run`) |
| **인증 관리** | macOS Keychain (폴백: 환경변수) |
| **리전** | ap-northeast-2 |

## 5. 성과 / 효과

### 응대 속도

- 문의 접수 즉시 **담당 조직에 자동 멘션** → 사람이 읽고 전달하는 단계 제거
- 업무 시간 외 문의도 알림이 즉시 도달

### 정확성

- 태그 11종 → 3개 조직 매핑으로 **오전달 감소**
- 미매핑 태그 폴백으로 **문의 누락 원천 차단**
- 중복 멘션 제거로 알림 피로도 완화

### 추적성

- 상담 접수~종료를 **하나의 Slack 스레드로 통합** → 맥락 유실 없이 이력 추적
- 상담 데이터 수집 체계로 **문의 유형 분석 및 CS 응대 매크로 설계** 근거 확보

---

## 6. 회고

**잘한 판단**

**폴백을 기본값으로 둔 것**. 라우팅 자동화에서 가장 위험한 실패는 오전달이 아니라 **미전달**입니다.
매핑 테이블에 없는 태그가 들어와도 기획팀에는 반드시 도달하게 설계해,
"자동화 때문에 문의가 사라지는" 상황을 구조적으로 없앴습니다.

그리고 **DynamoDB로 상담-스레드를 매핑한 것**. 상태를 갖지 않는 Lambda에서
두 개의 독립적인 Webhook 이벤트를 하나의 맥락으로 묶으려면 외부 상태 저장이 필요했고,
이 작은 테이블 하나가 Slack에서의 가독성을 크게 바꿨습니다.

**남은 과제**

- 태그 매핑이 코드에 하드코딩 → 조직 변경 시 재배포 필요 (설정 외부화 미적용)
- 라우팅 정확도·응대 시간 개선폭을 **정량 계측하지 못함**
