# Keeper Prototype - AI 기반 기획-프로토타이핑 사이클

> 텍스트 기획서 대신 "동작하는 프로토타입"으로 조기 검증하는 기획 파이프라인
> 프로젝트: `keeper-prototype` (React 19 / TypeScript / Vite)

---

## 프로젝트 개요

신규 기능을 기획할 때 **정책서와 동작하는 화면을 함께 만들어** 이해관계자가 개발 착수 전에 검증하도록 만든
피처(에픽)별 프로토타입 플랫폼입니다.

AI 코딩 도구 + [Keeper Agent](../11-keeper-agent/) 사내 지식을 결합해 도메인 파악 → 기획 초안 → 프로토타입 구현까지의
사이클을 압축했고, 사내 디자인 시스템 토큰과 실제 프로덕션 셸을 재현해 **프로덕션에 근접한 화면**으로 합의를 이끌어냅니다.

| 항목 | 내용 |
|------|------|
| **기간** | 2026.03 ~ 진행중 |
| **규모** | 115 커밋, 피처 5개 운영 (KP-1048/1230/1468/1521/1593) |
| **역할** | 기획(PO) + 프로토타입 구현 |
| **배포** | Vercel (이해관계자 공유 링크) |

---

## 1. 문제 정의

**문제 1 — 텍스트 기획서로는 인식이 안 맞는다**

- 신규 기능 기획 시 텍스트 기획서만으로는 이해관계자 간 인식 차이가 큼
- 개발 착수 후에야 "생각한 것과 다르다"가 드러나 **재작업 발생**

**문제 2 — 도메인 파악에 시간이 걸린다**

- 복잡한 사내 데이터 구조와 기존 시스템을 매번 사람이 파악해 기획 → 리드타임 증가

**문제 3 — 프로토타입이 실물과 동떨어지면 검증이 안 된다**

- 프로토타입마다 레이아웃·색상·용어가 제각각이면, 피드백이 "이게 실제로 저렇게 나오나요?"에서 멈춤
- 정책서와 화면이 분리되어 있으면 **화면을 보며 정책을 확인**할 수 없음

---

## 2. 문제 해결

### 2-1. 정책서와 화면을 한 화면에서 (Split Panel)

각 피처는 `policy/spec.md`(기능 명세 + 화면 설계)를 갖고, 이를 `?raw` import로 가져와
**화면 옆 split panel에 렌더링**합니다. 이해관계자는 화면을 조작하면서 동시에 근거 정책을 확인합니다.

```
┌─────────────────────────────┬──────────────────┐
│                             │  정책서 (spec.md)│
│   동작하는 프로토타입 화면   │  ├ 기능 명세     │
│                             │  ├ 화면 설계     │
│                             │  └ 비즈니스 규칙 │
└─────────────────────────────┴──────────────────┘
       react-markdown + remark-gfm + rehype-raw
```

### 2-2. ★ 핵심 설계 — 실물 재현 셸을 단일 기준(Single Source)으로

프로토타입이 늘어나면 **피처마다 레이아웃이 갈라지는 문제**가 필연적으로 생깁니다.
매번 JSX를 직접 짜면 어드민 사이드바 너비도, 앱 탭 순서도 조금씩 달라지고, 결국 "실물과 다른 화면"이 되어
검증 도구로서의 신뢰를 잃습니다.

그래서 **실제 프로덕션 구조를 재현한 공유 셸**을 만들고, 피처는 이를 조합만 하도록 규칙을 세웠습니다.

| 공유 컴포넌트 | 재현 대상 | 고정한 실물 값 |
|---------------|-----------|----------------|
| `<AdminShell>` | 어드민 웹 (`indicator` DesktopLayout) | 검정 상단바 + 아이콘 레일 64px + 서브메뉴 260px, 실제 메뉴 트리 |
| `<KeeperAppFrame>` | 키퍼 앱 (`keeper-two`) | 앱바 56px + 하단 5탭(잡기/예약/수행/완료/마이) |
| `<TicketDetailModal>` | 어드민 일감 상세 모달 | 8단계 상태 스텝퍼, 4개 탭, 탭 노출 조건, 사이드바 336px |
| `<PrototypeShell>` | 피처 페이지 공통 골격 | 헤더 + 본문 + 정책서 패널 |
| `<PolicyPanel>` | 정책서 split panel 단품 | Confluence 주석 제거·빈 정책서 처리 내장 |

> **원칙**: enum·순서·라벨 같은 **실물 고정값은 상수로 export**하고 임의 변경을 금지했습니다.
> (`TICKET_STATUS_STEPS`, `TICKET_TABS`, `getVisibleTicketTabs` 등)
> 실물 구조가 바뀌면 **공유 컴포넌트 한 곳만 고쳐 전 피처에 반영**됩니다.

셸은 **chrome만 제공**하고, 각 탭 컨텐츠·사이드바 카드·액션 버튼은 피처가 `children`/`sidebar`/`footer`
슬롯으로 주입합니다. 공통 골격은 강제하되 피처 고유 로직은 자유롭게 두는 구조입니다.

### 2-3. 피처 격리 규칙

여러 피처가 한 레포에서 동시에 진행되므로, **서로를 깨뜨리지 않도록** 명시적 규칙을 두었습니다.

```
src/features/KP-xxxx/
├── index.tsx          # 피처 메인 페이지
├── docs.tsx           # 정책서 전체 페이지 뷰어 (TOC 자동 파싱)
├── components/        # 피처 전용 컴포넌트
├── lib/               # 전용 로직·상태·mock 데이터
└── policy/
    ├── spec.md        # 기능 명세 + 화면 설계
    └── rules.yaml     # 비즈니스 규칙 (선택)
```

1. 피처 폴더 **밖의 파일 수정 금지**
2. 공유 `components/ui/`(shadcn)·`components/shared/`는 **import만 허용, 수정 금지**
3. 피처 전용 컴포넌트는 피처 폴더 안에 생성
4. **다른 피처 폴더의 코드 import 금지**

`/new-feature` 스킬로 이 구조를 스캐폴딩해, 규칙을 문서가 아니라 **도구로 강제**했습니다.

### 2-4. AI를 붙인 지점

| 단계 | AI 활용 |
|------|---------|
| **도메인 파악** | Keeper Agent MCP로 사내 정책·스키마·API를 자연어 조회 |
| **기획 초안** | 파악한 도메인 기반으로 기능 명세·화면 설계 초안 작성 |
| **프로토타입 구현** | AI 코딩 도구로 React 컴포넌트 구현 (격리 규칙을 CLAUDE.md로 학습) |
| **다국어 키 생성** | Figma MCP로 디자인에서 텍스트 추출 + 사내 키 네이밍 규칙 학습 → 키 자동 생성 |

## 4. 기술 스택

| 영역 | 기술 |
|------|------|
| **Framework** | React 19.2 + TypeScript 5.9 |
| **Build** | Vite 8 (HMR) |
| **Routing** | react-router v7 (중첩 라우트) |
| **UI** | shadcn/ui + @base-ui/react |
| **Styling** | Tailwind CSS v4 (`@tailwindcss/vite`) + 사내 디자인 토큰 |
| **Markdown** | react-markdown + remark-gfm + rehype-raw |
| **Icons** | lucide-react |
| **Lint** | ESLint 9 + typescript-eslint |
| **Deploy** | Vercel |
| **AI 연동** | Keeper Agent MCP, Figma MCP |

### 상태 관리 패턴

- 단순 상태 → 로컬 `useState`
- 복잡한 상태 → **Context Provider 패턴** (`{feature}-context.tsx` + `use{Feature}()` hook)
- Mock 데이터 → `lib/{도메인}/types.ts`(인터페이스) + `mock-data.ts`(named export 상수) 분리

## 5. 성과 / 효과

### 기획 품질

- 텍스트 기획서 대신 **동작하는 프로토타입으로 조기 검증**하는 사이클을 구축해, 기획-개발 간 인식 차이와 재작업 축소
- 정책서 split panel로 **화면과 근거를 동시에 확인** → 리뷰 시 "이 화면의 규칙이 뭔가" 왕복 질의 제거
- 5개 피처(에픽)를 이 사이클로 진행

### 일관성 / 유지보수

- 실물 재현 셸을 단일 기준으로 세워 **피처가 늘어도 화면 틀이 갈라지지 않음**
- 실물 구조 변경 시 공유 컴포넌트 한 곳 수정으로 전 피처 반영
- 피처 격리 규칙 + 스캐폴딩 스킬로 **동시 진행 시 충돌 방지**

### 속도

- 사내 디자인 시스템 토큰을 그대로 입혀 실제 화면에 가까운 프로토타입 → **이해관계자 합의 속도 향상**
- Keeper Agent 연동으로 도메인 파악 시간 단축
- Figma MCP + 키 규칙 학습으로 **다국어 키 생성 자동화**

## 6. 회고

**잘한 판단**

프로토타입을 "빨리 만들고 버리는 것"으로 두지 않고 **공유 셸에 투자**한 것. 초기에는 피처마다 화면을 새로 그리는 게
빨라 보였지만, 피처가 3개를 넘어가면서 틀이 갈라지기 시작했습니다. 실물 값을 상수로 고정하고 셸을 단일 기준으로
세운 뒤로는 새 피처의 시작 비용이 크게 줄었고, 무엇보다 **"실물과 같은가"라는 의심이 사라져** 검증 도구로 신뢰받게 됐습니다.

**남은 과제**

- 프로토타입 → 실제 구현 이관 시 코드 재사용은 제한적 (mock 기반이므로 의도된 것이나, 이관 비용은 존재)
- 합의 속도 향상을 정량 지표로 계측하지 못함
