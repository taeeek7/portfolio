# 서버 로그 Lakehouse 파이프라인

> 일 300만 건 서버 로그의 장기 보관·분석을 위한 GCS + BigQuery Lakehouse
> 프로젝트: `kafka`, `kube-airflow` (로그 수집·변환 도메인)

---

## 프로젝트 개요

기존 모니터링 툴의 짧은 보존 기간 때문에 불가능했던 **장기 로그 추적·트렌드 분석**을 가능하게 만든 파이프라인입니다.

`Server → Kafka(실시간 수집) → GCS(원본 보관) → Airflow(배치 변환) → BigQuery(분석)` 구조로
**실시간 수집과 배치 처리를 분리**해, 수집 안정성과 분석 비용을 동시에 확보했습니다.

| 항목 | 내용 |
|------|------|
| **기간** | 2025.08 ~ 2025.10 |
| **처리 규모** | 일 300만 건 |
| **대상** | 3종 서버(배치·어드민·앱 백엔드) × 2개 환경(dev/prod) |
| **역할** | 아키텍처 설계 및 단독 구축 |

---

## 1. 문제 정의

**문제 1 — 로그가 오래 남지 않아 근본 원인 분석이 불가능**

- 기존 모니터링 툴의 짧은 로그 보존 기간으로, 장애 발생 시 **과거 시점 추적이 불가능**
- 장기 트렌드 기반 디버깅(언제부터 이 에러가 늘었나)을 할 수 없음

**문제 2 — 로그와 운영 데이터가 분리되어 있음**

- 서버 로그가 BigQuery 분석 환경과 분리 → 운영 데이터와 **통합 분석하거나 학습 데이터로 활용 불가**

**문제 3 — 그냥 다 넣으면 비용이 감당 안 됨**

- 일 300만 건을 그대로 웨어하우스에 적재하면 **스토리지·스캔 비용이 비선형적으로 증가**

---

## 2. 문제 해결

### 2-1. ★ 아키텍처 선정 — 4개 축으로 비교 후 결정

"로그를 어디에 쌓을 것인가"는 되돌리기 어려운 결정이라, 후보를 **명시적 기준으로 비교**한 뒤 선택했습니다.

| 비교 축 | 검토 내용 |
|---------|-----------|
| **적재량** | 일 300만 건 × 장기 보관 시 증가 곡선 |
| **비용** | 스토리지 단가 + 조회 시 스캔 과금 구조 |
| **관리형 서비스 복잡도** | 운영 인력이 사실상 1명인 환경에서 감당 가능한가 |
| **기존 분석 환경과의 연계성** | 이미 BigQuery에 운영 데이터가 있음 |

| 후보 | 판단 |
|------|------|
| NCP Data Query | 관리 부담은 낮으나 **기존 BigQuery 분석 환경과 분리** → 통합 분석 목적에 어긋남 |
| 전량 BigQuery 직접 적재 | 연계성은 최고이나 **원본 장기 보관 비용이 과다** |
| **GCS + BigQuery Lakehouse (채택)** | 원본은 **저렴한 오브젝트 스토리지**에, 분석은 **기존 웨어하우스**에서. 두 요구를 모두 충족 |

> **핵심 판단**: 로그는 "대부분 안 보지만 필요할 때 반드시 있어야 하는" 데이터입니다.
> 이 성격에는 **저장은 싸게, 조회는 필요한 만큼만** 과금되는 Lakehouse가 맞다고 봤습니다.

### 2-2. 실시간 수집과 배치 처리의 분리

```
Server ──▶ Kafka ──▶ GCS (원본 JSON.gz)
           실시간         │
                          │ Airflow 배치 (매일 새벽)
                          ▼
                    GCS (Parquet, Hive 파티션)
                          │
                          ▼
                    BigQuery 외부 테이블
```

수집과 변환을 분리한 이유:

- **수집 안정성** — 변환 로직 실패가 로그 유실로 이어지지 않음. 원본은 이미 GCS에 안전하게 있음
- **재처리 가능** — 변환에 문제가 있으면 원본에서 다시 돌리면 됨
- **비용** — 실시간 변환은 비싸지만, 로그 분석은 대부분 D-1이면 충분

매니지드 Kafka 위에 **토픽·컨슈머 구성을 직접 설계·운영**해 실시간 수집 환경을 확보했습니다.

### 2-3. ★ 비용 최적화 — Parquet 변환 + Hive 파티셔닝

BigQuery는 **스캔한 바이트 수로 과금**됩니다. 이 과금 구조에 맞춰 저장 포맷을 설계했습니다.

| 최적화 | 효과 |
|--------|------|
| **JSON.gz → Parquet (컬럼형)** | 필요한 컬럼만 스캔 → 전체 행을 읽지 않음 |
| **Hive 파티션** `year=YYYY/month=MM/day=DD` | 날짜 조건으로 **파티션 프루닝** → 조회 범위 밖 파일을 아예 안 읽음 |
| **BigQuery 외부 테이블** | GCS에 둔 채로 조회 → 웨어하우스 **중복 적재 비용 없음** |

```python
# 원본 경로 (수집 시점 구조)
source_prefix = f"prod-{server_type}-logs/{year}/{month}/{day}/"

# 대상 경로 (Hive 파티션 — BigQuery 외부 테이블용)
target_prefix = f"prod-{server_type}-logs/year={year}/month={month}/day={day}/"
```

> 수집 경로(`/2026/07/21/`)와 분석 경로(`year=2026/month=07/day=21/`)를 **분리**한 것이 포인트입니다.
> 수집 측은 단순한 구조를 유지하고, 분석에 필요한 파티션 규약은 변환 단계에서 입힙니다.

### 2-4. 운영 안정성

```python
default_args={
    'on_failure_callback': notify_failure,      # Slack 실패 알림
    'retries': 2,
    'retry_delay': timedelta(minutes=5),
    'execution_timeout': timedelta(hours=1),    # 무한 점유 방지
}
```

- **환경 분기** — `PROPERTY_ENV` Variable로 dev/live를 분리, 알림 채널도 환경별로 라우팅
- **D-1 기준 처리** — KST 기준 전날 데이터를 처리해 타임존 경계 문제 회피
- **순차 실행** — 3개 서버 타입을 순차 처리해 GCS API 부하 분산
- **적재량 모니터링** — Kafka → BigQuery 싱크 헬스체크 DAG를 별도 운영해 파이프라인 장애 조기 감지

## 3. 아키텍처

```
┌──────────────────────────────────────────────────────────────┐
│  3종 서버 (배치 / 어드민 / 앱 백엔드) × 2개 환경(dev·prod)    │
└────────────────────────────┬─────────────────────────────────┘
                             │ 애플리케이션 로그
                             ▼
                    ┌─────────────────┐
                    │  Kafka Topics   │  실시간 수집
                    │  {env}-{svc}-   │  (토픽·컨슈머 직접 설계)
                    │      logs       │
                    └────────┬────────┘
                             ▼
                    ┌─────────────────────────────┐
                    │  GCS 원본 버킷               │
                    │  {env}-server-logs           │
                    │  └ {svc}-logs/YYYY/MM/DD/    │
                    │      *.json.gz               │
                    └────────┬────────────────────┘
                             │
                    ┌────────▼─────────────────────────────┐
                    │  Airflow DAG (매일 새벽, KST)         │
                    │  transform_{env}_server_log_in_gcs    │
                    │                                       │
                    │  batch ──▶ indicator ──▶ keeper       │
                    │  (순차 실행, 각 D-1 로그 변환)        │
                    │                                       │
                    │  JSON.gz ──▶ Parquet (PyArrow)        │
                    └────────┬─────────────────────────────┘
                             ▼
                    ┌─────────────────────────────┐
                    │  GCS 변환 버킷               │
                    │  transformed-server-logs/    │
                    │  └ {svc}-logs/               │
                    │     year=/month=/day=/       │
                    │       *.parquet              │
                    └────────┬────────────────────┘
                             ▼
                    ┌─────────────────────────────┐
                    │  BigQuery 외부 테이블        │
                    │  + 운영 데이터 통합 분석     │
                    └─────────────────────────────┘
                             │
                             ▼
                    실패 시 Slack 알림 (환경별 채널)
```

## 4. 기술 스택

| 영역 | 기술 |
|------|------|
| **수집** | Apache Kafka (매니지드, 토픽·컨슈머 직접 구성) |
| **오케스트레이션** | Apache Airflow (K8s, `PythonOperator`) |
| **스토리지** | Google Cloud Storage (원본 / 변환 버킷 분리) |
| **포맷 변환** | Parquet (PyArrow) |
| **웨어하우스** | BigQuery (외부 테이블 + Hive 파티션) |
| **알림** | Slack (환경별 채널 분기) |
| **언어** | Python 3.11 |
| **스케줄** | cron (`40 1 * * *`, KST), `catchup=False` |

## 5. 성과 / 효과

### 관측 가능성

- 일 300만 건 규모 로그를 **안정적으로 장기 보관** → 과거 시점까지 추적 가능한 디버깅 환경 확보
- 기존 모니터링 툴의 보존 기간 제약 해소 → **장기 트렌드 기반 원인 분석** 가능

### 비용 / 성능

- JSON → **Parquet 컬럼형 변환**으로 필요 컬럼만 스캔 → BigQuery 쿼리 비용·조회 시간 개선
- **Hive 파티셔닝**으로 날짜 범위 밖 파일을 스캔에서 제외 (파티션 프루닝)
- 외부 테이블 활용으로 **웨어하우스 중복 적재 비용 제거**

### 분석 기반

- 로그 데이터와 운영 데이터를 **동일 환경에서 통합 분석** 가능 → BI·분석 기반 마련
- 원본을 GCS에 온전히 보존해 **향후 학습 데이터로 재활용 가능한 상태** 유지

### 운영

- Airflow 실패 알람 + 적재량 모니터링으로 **파이프라인 장애 조기 감지**
- 원본/변환 버킷 분리로 **변환 실패 시 재처리 가능** (로그 유실 없음)

---

## 6. 회고

**잘한 판단**

**수집과 변환을 분리한 것**. 초기에는 Kafka에서 바로 BigQuery로 흘리는 안도 검토했지만,
그러면 변환 로직에 문제가 생겼을 때 로그 자체가 유실됩니다. 원본을 GCS에 그대로 떨어뜨리고
변환은 배치로 분리하니, **재처리가 언제든 가능한 구조**가 되어 운영 부담이 크게 줄었습니다.

그리고 아키텍처를 **4개 축으로 비교하고 근거를 남긴 것**. "왜 NCP Data Query가 아니라 GCS+BigQuery인가"를
설명할 수 있어야 나중에 조건이 바뀌었을 때(예: 분석 환경이 옮겨갈 때) 재검토가 가능합니다.

**남은 과제**

- 비용 절감을 **절감률 수치로 계측하지 못함** (변환 전후 대조 데이터 미확보)
- 로그 스키마 변경 시 Parquet 변환이 깨질 수 있음 — 스키마 진화 대응 미구현
