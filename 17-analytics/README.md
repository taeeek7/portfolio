# 프로덕트 애널리틱스 - GA4 이벤트 설계 & 운영 대시보드

> 가설 검증을 위한 이벤트 트래킹 설계 + BigQuery 마트 기반 전사 대시보드
> 프로젝트: `integrations/{google-analytics,redash,databases}`

---

## 프로젝트 개요

신규 어드민 대시보드의 **사용성을 데이터로 검증**하기 위한 GA4 커스텀 이벤트 체계를 설계하고,
BigQuery 마트를 기반으로 **전사 운영 현황 대시보드**를 구축한 프로젝트입니다.

"이 화면이 정말 편한가?"라는 질문을 **검증 가능한 가설과 지표로 번역**하는 데 초점을 맞췄습니다.

| 항목 | 내용 |
|------|------|
| **기간** | 2026.05 ~ |
| **역할** | 이벤트 트래킹 설계(PO), 대시보드 설계·구축 |
| **산출물** | GA4 이벤트 명세서, BQ 가설검증 쿼리, 전사 운영 대시보드 |

---

## 1. 문제 정의

**문제 1 — 개선했다고 말할 근거가 없다**

- 대시보드 페이지를 새로 만들었지만, 기존 페이지 대비 **정말 나아졌는지 확인할 방법이 없음**
- "좋아진 것 같다"는 인상 외에 대표 요청에 답할 **사실 기반 근거 부재**

**문제 2 — 이벤트를 그냥 심으면 나중에 해석할 수 없다**

- 목적 없이 클릭 이벤트만 잔뜩 수집하면, 정작 질문이 생겼을 때 답을 못 냄
- **무엇을 검증할지 먼저 정하지 않으면** 수집한 데이터가 쓸모없어짐

**문제 3 — 운영 현황이 각자의 화면에만 있다**

- 전사 차원의 운영 현황(처리량·매출·성장세)을 한눈에 볼 공용 화면이 없음

## 2. 문제 해결

### 2-1. ★ 핵심 접근 — 가설 → 지표 → 이벤트 순으로 역설계

이벤트를 먼저 심지 않고, **검증할 가설부터 정의**한 뒤 필요한 이벤트를 역산했습니다.

```
[가설]  대시보드 안에서 작업을 완결할 수 있다 (자족성)
   ↓
[지표]  자족성 = 모달 진입·타 페이지 이탈이 적을수록 높다
   ↓
[이벤트] card_detail_open, *_status_change(via_modal)
```

**가설별 지표 매핑 예시 (H1 자족성)**

| 지표 | 방향 | 해석 |
|------|------|------|
| 세션당 `card_detail_open` 빈도 | ↑일수록 자족성 **↓** | 카드만으로 부족해 모달에 의존 |
| `*_status_change.via_modal=true` 비율 | ↑일수록 자족성 **↓** | 상태 변경을 카드에서 못 끝냄 |

> **설계 의도**: 디자인 의도가 "여기서 모든 걸 끝내라"였으므로,
> **모달 진입과 타 페이지 이탈을 역지표로 정의**했습니다.
> 지표의 방향(↑가 좋은가 나쁜가)을 명세서에 명시해, 나중에 해석이 뒤집히지 않게 했습니다.

검증 가설은 성격을 나눠 세웠습니다.

| ID | 가설 | 비교 기준 |
|----|------|-----------|
| H1 | 대시보드 안에서 작업을 완결할 수 있다 | 모달/타 페이지 의존도 |
| H3 | 미해결 이슈·미확인 클레임 평균 잔존 시간이 줄어든다 | **출시 전후 비교** |

H1은 화면 내부 행동으로, H3은 **운영 결과 지표의 전후 비교**로 검증하도록 설계해
"쓰기 편한가"와 "실제로 일이 빨라졌나"를 분리했습니다.

### 2-2. 명세서를 협업 단일 창구로

이벤트 명세서를 **Figma 디자인 + 개발 작업 시트 + Confluence와 연결**해 관리했습니다.

- 문서에 버전(`0.10`)과 최종 업데이트 일자를 명시
- `confluence-sync` 메타데이터로 **로컬 마크다운 ↔ Confluence 양방향 동기화**
- 도메인 용어(ticket=일감, report=보고사항, claim=클레임, room_group=지점)를 문서 상단에 정의해
  **개발자와 기획자가 같은 이름을 쓰도록** 강제

### 2-3. 전사 운영 대시보드 — 데이터 함정을 문서화

사무실 대형 화면에 상시 노출하는 운영 현황 대시보드를 BigQuery 마트 기반으로 구축했습니다.

**부딪힌 데이터 품질 문제와 대응**

| 문제 | 대응 |
|------|------|
| `updated_at`이 "완료 시각"이 아니라 "마지막 업데이트 시각" — 완료 후 메모 수정 시 갱신됨 | 시간대 분포를 분석해 **11~14시 폭증 패턴(체크아웃 청소 완료)** 확인 → 근사값으로 사용 가능 판단, 한계를 문서에 명시 |
| `ETC` 타입에 호텔·생활형숙박·사무실이 섞여 있음 | PMS 타입 조합으로 **재분류 CASE WHEN** 적용, 근본 해결은 마스터 데이터 정리로 별도 요청 |
| DATETIME은 UTC 저장, `resolved_at`은 KST 문자열 | 컬럼별 타입·타임존 규약을 표로 정리해 **쿼리마다 일관 적용** |
| 매출 컬럼이 `_amount` / `_confirmed_amount` 두 가지 | 현재 값은 동일하나 **미래 호환을 위해 `_confirmed_amount` 우선** 채택, NULL은 `COALESCE`로 0 처리 |

> 데이터 함정을 **디자인 문서에 명시적으로 남긴 것**이 핵심입니다.
> 특히 `updated_at`은 엄밀하지 않은 근사값인데, 이를 숨기지 않고
> "왜 그럼에도 쓸 수 있다고 판단했는지"를 근거와 함께 기록했습니다.

### 2-4. BigQuery Export 연동

GA4 데이터를 BigQuery로 export해 **SQL 기반 심화 분석**이 가능하도록 구성하고,
가설검증용 쿼리를 별도로 작성했습니다.

## 5. 성과 / 효과

### 의사결정 근거

- "대시보드가 편한가"를 **검증 가능한 가설 2종 + 지표 매핑**으로 번역 → 출시 후 사실 기반 판정 가능
- 지표의 방향(역지표 포함)을 명세에 고정해 **해석 논쟁 제거**
- 대표 요청에 대한 **데이터 기반 답변** 마련

### 데이터 신뢰성

- 마트 컬럼의 타입·타임존·의미 규약을 문서화해 **쿼리 간 일관성 확보**
- `updated_at` 근사 사용, `ETC` 재분류 등 **데이터 한계를 숨기지 않고 명시** → 잘못된 해석 방지
- 마스터 데이터 정리 필요 항목을 식별해 **근본 해결 과제로 별도 제기**

### 협업

- 이벤트 명세서를 Figma·개발 시트·Confluence와 연결해 **단일 창구화**
- 도메인 용어 정의를 문서 상단에 두어 **기획-개발 간 용어 불일치 제거**
- GA4 설정을 스크립트로 자동 적용해 **수기 설정 실수 방지**

### 가시성

- 전사 운영 현황(처리량·매출·성장세·시간대 패턴)을 **사무실 대형 화면에 상시 노출**

## 6. 회고

**잘한 판단**

**가설을 먼저 세우고 이벤트를 역산한 것**. 트래킹은 일단 많이 심어두고 나중에 보자는 유혹이 크지만,
목적 없이 쌓인 이벤트는 질문이 생겼을 때 대체로 답을 못 줍니다.
"자족성"처럼 추상적인 개념을 **모달 진입률이라는 역지표로 조작화**한 과정이 이 프로젝트의 핵심이었습니다.

그리고 **데이터 함정을 문서에 남긴 것**. `updated_at`이 엄밀한 완료 시각이 아니라는 걸 알면서도 쓴 이유와
그 판단 근거(시간대 분포 분석)를 기록해두니, 나중에 이 대시보드를 보는 사람이
숫자를 **어디까지 믿어야 하는지** 알 수 있게 됐습니다.

**남은 과제**

- H3(잔존 시간 감소)는 **출시 전후 비교가 필요해 아직 검증 미완료**
- GA4 데이터 기반 자연어 질의 에이전트는 장기 목표로만 남아 있음
- 근본 원인인 `room_group.type` 마스터 데이터 정리는 미해결 (재분류로 우회 중)
